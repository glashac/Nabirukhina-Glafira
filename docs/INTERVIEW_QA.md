## 1. Общие вопросы о проекте

### В1: Расскажите о постановке задачи курсовой работы.
**А1:** 
Целью курсовой работы было разработать клиент-серверное приложение для управления арендой автомобилей. Приложение должно реализовать трёхзвенную архитектуру (клиент-сервер-БД), обеспечить CRUD-операции, фильтрацию, сортировку и статистику в соответствии с методическими указаниями по дисциплине «Современные технологии программирования». Я использовал Spring Boot для сервера, JavaFX для клиента и JPA/Hibernate для работы с БД H2.

### В2: Какие технологии вы использовали и почему их выбрали?
**А2:**
- **JDK 17**: современная версия Java с лучшей производительностью
- **Spring Boot 3**: быстрая разработка REST API, встроенный контейнер Tomcat
- **JavaFX**: кросс-платформенный UI для десктопа (лучше, чем Swing)
- **JPA/Hibernate**: стандарт ORM в Java для автоматического маппинга объектов в БД
- **H2**: лёгкая in-memory БД для прототипирования
- **Maven**: стандартный инструмент управления зависимостями в Java

### В3: Соответствует ли проект требованиям методички?
**А3:**
Да, проект полностью соответствует всем требованиям методички:
- Трёхзвенная архитектура (клиент ↔ сервер ↔ БД)
- ORM (JPA/Hibernate)
- GUI (JavaFX с таблицей и формами)
- CRUD операции (добавить, просмотреть, редактировать, удалить)
- Фильтрация (по модели и типу)
- Сортировка (по любой колонке в TableView)
- Статистика (среднее, минимум, максимум цены)

- JavaDoc документация

## 2. Вопросы об архитектуре

### В4: Расскажите об архитектуре приложения.
**А4:**
Приложение построено по трёхзвенной архитектуре:

1. **Клиентский слой** (JavaFX):
   - MainApp — главное окно с меню, панелью инструментов и таблицей
   - CarFormDialog — диалог для добавления/редактирования
   - CarApi — HTTP-клиент для общения с сервером

2. **Серверный слой** (Spring Boot):
   - CarController — REST-контроллер с 6 эндпоинтами (CRUD + статистика)
   - CarRepository — интерфейс для работы с БД (JPA)
   - DataInitializer — инициализация демо-данных

3. **Слой данных** (H2 + JPA):
   - Car (JPA Entity) — сущность автомобиля
   - Таблица `cars` с 6 полями
   - Автоматическое создание схемы через hibernate.ddl-auto=create-drop

Клиент отправляет HTTP-запросы (JSON) на сервер, сервер обрабатывает через Repository и возвращает JSON-ответы.

### В5: Почему вы выбрали именно эту архитектуру?
**А5:**
Трёхзвенная архитектура:
- **Разделение ответственности**: каждый слой отвечает за свою задачу
- **Масштабируемость**: легко заменить H2 на PostgreSQL без изменения клиента
- **Переиспользование**: сервер может использоваться с другими клиентами (мобильное приложение, веб и т.д.)
- **Соответствие методичке**: это требование для оценки «отлично»
- **Стандартный паттерн**: используется в production-коде

### В6: Какие паттерны проектирования вы использовали?
**А6:**
- **MVC** (Model-View-Controller) в сервере
- **DAO** (Data Access Object) — CarRepository
- **DTO** неявный через JSON сериализацию
- **Singleton** — Spring Beans для CarApi, CarRepository
- **Dependency Injection** — Spring автоматически инжектит зависимости

## 3. Вопросы о сервере и API

### В7: Расскажите о REST API сервера.
**А7:**
REST API имеет 6 основных эндпоинтов:
- `GET /api/cars` — получить список (с фильтрами ?q=модель или ?type=тип)
- `GET /api/cars/{id}` — получить по ID
- `POST /api/cars` — создать новый (JSON body)
- `PUT /api/cars/{id}` — обновить (JSON body)
- `DELETE /api/cars/{id}` — удалить
- `GET /api/cars/stats` — получить статистику (count, avg, min, max цена)

Все эндпоинты возвращают JSON. Коды ответов: 200 OK, 201 Created, 204 No Content, 404 Not Found, 500 Error.

### В8: Как реализована фильтрация?
**А8:**
Фильтрация реализована на сервере через CarRepository:
- `findByModelContainingIgnoreCase(String model)` — поиск по модели (case-insensitive)
- `findByTypeContainingIgnoreCase(String type)` — поиск по типу

В контроллере:
```java
if (q != null && !q.isBlank()) {
    return carRepository.findByModelContainingIgnoreCase(q);
}
```

Клиент отправляет параметр `?q=velocity`, сервер фильтрует данные через LIKE запрос в H2.

### В9: Как работает статистика?
**А9:**
Статистика вычисляется на сервере с помощью Stream API:
```java
List<Car> all = carRepository.findAll();
double avg = all.stream().mapToDouble(Car::getPricePerDay).average().orElse(0);
double min = all.stream().mapToDouble(Car::getPricePerDay).min().orElse(0);
double max = all.stream().mapToDouble(Car::getPricePerDay).max().orElse(0);
```

Возвращается DTO (record): `Stats(count, avgPrice, minPrice, maxPrice)`.

В клиенте нажимаем "Обновить статистику" → запрос на `/api/cars/stats` → отображение в Label.

### В10: Как обрабатываются ошибки на сервере?
**А10:**
- **404 Not Found**: если автомобиль не существует
  ```java
  return carRepository.findById(id)
      .map(ResponseEntity::ok)
      .orElseGet(() -> ResponseEntity.notFound().build());
  ```
- **400 Bad Request**: если JSON неверный формат
- **500 Internal Server Error**: если ошибка в базе данных
- Логирование через SLF4J в консоль

## 5. Вопросы о коде и ООП

### В15: Как вы использовали ООП?
**А15:**
- **Инкапсуляция**: поля private, доступ через getter/setter
- **Наследование**: 
  - Server Car наследует методы JpaRepository
  - Client Car использует JavaFX Properties
- **Полиморфизм**: HttpClient используется через интерфейс
- **Абстракция**: CarRepository как интерфейс для доступа к БД

Пример инкапсуляции:
```java
public class Car {
    private Long id;
    private String model;
    
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
}
```

### В17: Как вы обрабатываете исключения?
**А17:**
- **На сервере**: try-catch в контроллерах, возврат HTTP-статусов
- **На клиенте**: try-catch в CarApi (для сетевых ошибок), Alert-диалоги для пользователя
- **Валидация на клиент-стороне**: проверка перед отправкой на сервер
- **Логирование**: все ошибки логируются через SLF4J

Программа **не завершается аварийно** — все ошибки обрабатываются корректно с информативными сообщениями.

## 6. Вопросы о БД и JPA

### В18: Расскажите о базе данных.
**А18:**
Используется **H2 Database** (in-memory):
- Быстрая для разработки и тестирования
- При запуске сервера автоматически создаётся таблица `cars`
- Демо-данные инициализируются через DataInitializer

Таблица `cars`:
```sql
CREATE TABLE cars (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  model VARCHAR(255),
  type VARCHAR(255),
  year INT,
  price_per_day DOUBLE,
  available BOOLEAN
);
```

### В19: Как вы использовали ORM (JPA)?
**А19:**
JPA используется для:
- **Автоматического маппинга** объектов Car на таблицу cars
- **CRUD операций** через методы Repository (save, findById, delete)
- **Кастомных запросов** (findByModelContainingIgnoreCase)

Конфигурация в application.properties:
```properties
spring.jpa.hibernate.ddl-auto=create-drop  # Автосоздание схемы
spring.datasource.url=jdbc:h2:mem:car_rental
```

**Преимущества ORM:**
- Не нужно писать SQL вручную
- Автоматическая валидация и маппинг типов
- Легче менять БД (H2 → PostgreSQL)

### В20: Как данные сохраняются между запусками?
**А20:**
В текущей версии используется **in-memory H2** → данные теряются при перезагрузке сервера.

Для продакшена нужно:
1. Заменить на **файловую БД** или **PostgreSQL**
2. Изменить connection string в application.properties:
   ```properties
   spring.datasource.url=jdbc:postgresql://localhost:5432/car_rental
   spring.datasource.username=postgres
   spring.datasource.password=password
   ```

Данные будут сохраняться в файле или в БД.

## 8. Сложные вопросы (6+ баллов)

### В23: Какие известные вам недостатки проекта и как их исправить?
**А23:**
**Недостатки:**
1. **Нет аутентификации** → добавить Spring Security + JWT
2. **H2 in-memory** → заменить на PostgreSQL для persistence
3. **Нет кэширования** → добавить Redis для часто запрашиваемых данных
4. **Нет WebSocket** → для real-time обновлений от сервера
5. **Нет интеграционных тестов** → добавить JUnit + Testcontainers

**Как исправить:**
- Расширить функциональность без изменения архитектуры
- Следовать SOLID принципам
- Добавить слой Service между Controller и Repository

### В24: Как масштабировать приложение на продакшен?
**А24:**
1. **Backend:**
   - Заменить H2 на PostgreSQL (или другой production-ready RDBMS)
   - Добавить connection pooling (HikariCP, уже встроено в Spring)
   - Кэширование (Redis)
   - Message Queue (Kafka) для асинхронных операций
   - Docker контейнеризация

2. **Frontend:**
   - Перейти на веб-приложение (React/Angular/Vue)
   - Или оставить JavaFX но добавить мобильный клиент

3. **DevOps:**
   - CI/CD через GitHub Actions
   - Мониторинг через Prometheus/Grafana
   - Логирование через ELK stack

### В25: Почему вы выбрали именно H2, а не PostgreSQL маркает с самого начала?
**А25:**
Я выбрал H2 для MVP потому что:
- **Скорость разработки**: no setup needed, work out of the box
- **Простота**: one JAR file, no external DB needed
- **Тестирование**: fast in-memory, easy to reset between tests
- **Методичка**: требует ORM, не требует specific DBMS

Для курсовой работы это ideal для демонстрации. В production я бы использовал PostgreSQL с миграциями через Liquibase или Flyway.
